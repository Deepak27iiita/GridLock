1. EXECUTIVE SUMMARY
================================================================================

GridLock is an end-to-end AI parking intelligence platform built for Bengaluru
traffic enforcement. It addresses the hackathon problem:

  "How can AI-driven parking intelligence detect illegal parking hotspots and
   quantify their impact on traffic flow to enable targeted enforcement?"

The system ingests 298,450 real police parking violation records, processes
them through an ML pipeline, detects 60 priority hotspots (40 named junctions +
20 corridor zones), scores each zone with a Parking Congestion Impact Score
(PCIS), forecasts violation patterns by hour, and generates patrol deployment
plans with estimated delay savings.

Deliverables implemented:
  - Full ML training pipeline (scripts/train_pipeline.py)
  - Fast rescore utility (scripts/rescore_pcis.py)
  - REST API with 12 endpoints (FastAPI)
  - Live command-center dashboard with heatmap, charts, and simulator
  - Model evaluation with published accuracy metrics


================================================================================
2. PROBLEM STATEMENT & SOLUTION DIRECTION
================================================================================

OPERATIONAL CHALLENGE:
  On-street illegal parking and spillover parking near commercial areas, metro
  stations, and events choke carriageways and intersections.

WHY IT IS HARD TODAY:
  - Enforcement is patrol-based and reactive
  - No heatmap of parking violations vs. congestion impact
  - Difficult to prioritize enforcement zones

GRIDLOCK SOLUTION:
  1. Detect hotspots automatically from historical violation GPS data
  2. Score each hotspot with PCIS (0–1) based on violation density, severity,
     junction proximity, peak-hour congestion, and lane spillover risk
  3. Rank zones for enforcement priority
  4. Recommend time-windowed patrol deployments
  5. Simulate impact of additional enforcement teams
  6. Visualize everything on an interactive dashboard

6. DATA PREPROCESSING PIPELINE
================================================================================

FILE: backend/app/services/preprocess.py
ENTRY: scripts/train_pipeline.py (Step 1–2)

STEP-BY-STEP PROCESS:

6.1 LOAD RAW DATA
  - Reads CSV with memory-optimized column selection (11 columns only)
  - Avoids loading unused fields to prevent out-of-memory errors

6.2 QUALITY FILTERING
  - Removes records with validation_status = "rejected"
  - Retains: approved, created1, processing, and unvalidated records
  - Result: ~242,362 clean records

6.3 VIOLATION TYPE PARSING
  - Parses JSON array in violation_type column
  - Fallback regex extraction for malformed JSON
  - Extracts primary_violation (first type in list)
  - Computes severity_weight using VIOLATION_WEIGHTS config

6.4 VEHICLE SIZE FACTOR
  - Maps vehicle_type to congestion multiplier
  - TANKER=1.25, BUS=1.30, CAR=1.0, SCOOTER=0.70, etc.
  - composite_severity = severity_weight × vehicle_factor

6.5 TEMPORAL FEATURES
  - Converts created_datetime to UTC, then IST (UTC+5:30)
  - hour_ist (0–23), day_of_week (0=Mon), is_weekend, month

6.6 JUNCTION FLAGS
  - is_junction = 1 if junction_name is not "No Junction"
  - junction_label = cleaned junction name

6.7 H3 GEOSPATIAL INDEXING
  - Assigns H3 cell index at resolution 9 (~174m hexagons)
  - Uses h3.latlng_to_cell(lat, lon, 9) for each record
  - Enables spatial aggregation and heatmap generation

6.8 DEDUPLICATION
  - dedupe_key = vehicle_number + h3_index + 15-minute time bucket
  - Keeps first occurrence only
  - Prevents double-counting same parked vehicle

6.9 CELL FEATURE AGGREGATION
  - Groups violations by h3_index
  - Computes per-cell: violation_count, avg_severity, junction_ratio,
    peak_hour_share, dominant_violation, top_station, top_junction
  - Derives: violation_density, spillover_risk, temporal_congestion,
    junction_proximity, severity_score

6.10 SAVE PROCESSED DATA
  - violations.parquet — Slim violation records with features
  - cells.parquet — H3 cell aggregated features


================================================================================
7. GEOSPATIAL INDEXING (H3)
================================================================================

LIBRARY: h3-py 4.1.2
RESOLUTION: 9 (configured in backend/app/config.py as H3_RESOLUTION = 9)

WHY H3:
  - Uber's hexagonal hierarchical geospatial index
  - Resolution 9 ≈ 174m edge length — ideal for street-level parking zones
  - Enables uniform spatial aggregation unlike arbitrary lat/lon grids
  - Supports heatmap rendering at consistent cell sizes

OUTPUT:
  - 2,426 unique H3 cells across Bengaluru from violation data
  - Each cell has latitude/longitude centroid, PCIS score, violation count

8. HOTSPOT DETECTION ENGINE
================================================================================

FILE: backend/app/services/clustering.py

APPROACH: Hybrid two-tier hotspot detection

TIER 1 — JUNCTION HOTSPOTS (40 zones):
  - Groups violations where is_junction = 1
  - Aggregates by junction_label (e.g., "BTP051 - Safina Plaza Junction")
  - Ranks by violation_count descending
  - Takes top 40 named junctions
  - Each hotspot includes: centroid lat/lon, violation count, dominant
    violation type, police station, list of H3 indexes in zone

TIER 2 — CORRIDOR HOTSPOTS (20 zones):
  - Identifies H3 cells NOT already covered by junction hotspots
  - Selects top 20 cells by violation_count
  - Names hotspot from most common location string
  - Captures non-junction parking corridors (ORR, commercial roads)

CLUSTERING VALIDATION (Silhouette Score):
  - Runs DBSCAN eps grid search on top 2,000 active H3 cells
  - Also runs KMeans (k=10,15,20,25) for comparison
  - Reports best silhouette score as clustering quality metric
  - Actual result: Silhouette = 0.2729

OUTPUT:
  - hotspots.parquet — 60 hotspots with metadata
  - dbscan.joblib — Clustering model artifact

TOP DETECTED HOTSPOTS:
  1. BTP051 - Safina Plaza Junction     — 11,754 violations
  2. BTP082 - KR Market Junction        —  9,484 violations
  3. BTP040 - Elite Junction            —  9,265 violations
  4. BTP044 - Sagar Theatre Junction    —  8,773 violations
  5. Kadubisanahalli corridor           —  5,340 violations


================================================================================
9. PCIS — PARKING CONGESTION IMPACT SCORE
================================================================================

FILE: backend/app/services/pcis.py

PURPOSE:
  Quantify how much each parking hotspot degrades traffic flow on a 0–1 scale.

9.1 RULE-BASED PCIS (Baseline)
  Formula:
    PCIS = 0.28 × violation_density
         + 0.22 × severity_score
         + 0.20 × junction_proximity
         + 0.18 × temporal_congestion
         + 0.12 × spillover_risk

  Violation Severity Weights (backend/app/config.py):
    DOUBLE PARKING                          = 1.00
    PARKING IN A MAIN ROAD                  = 0.95
    PARKING NEAR ROAD CROSSING              = 0.90
    PARKING NEAR TRAFFIC LIGHT              = 0.85
    NO PARKING                              = 0.70
    WRONG PARKING                           = 0.65
    PARKING ON FOOTPATH                     = 0.40

9.2 ML-ENHANCED PCIS (LightGBM + Isotonic Calibration)
  Features used:
    - violation_density, severity_score, junction_proximity
    - temporal_congestion, spillover_risk, log_violations, rule_pcis

  Model: LightGBM Regressor
    - n_estimators=250, learning_rate=0.05, max_depth=5

  Calibration: Isotonic Regression on hold-out test set
    - Ensures predicted scores are well-calibrated to actual impact

  Blending: 45% ML prediction + 55% rule-based impact target

9.3 PERCENTILE SPREADING (Score Recalibration)
  - Applies rank percentile transformation: score^0.68
  - Spreads scores across full 0–1 range
  - Ensures top hotspots reach Severe/Critical levels
  - Implemented in scripts/rescore_pcis.py for fast updates

9.4 DELAY ESTIMATION
  estimated_delay_min_per_hour =
    PCIS × sqrt(violation_count) × BASE_DELAY_PER_VIOLATION × 1.15

  BASE_DELAY_PER_VIOLATION = 0.82 minutes (config constant)

9.5 RISK LEVEL CLASSIFICATION
  PCIS >= 0.85  →  Severe
  PCIS >= 0.72  →  Critical
  PCIS >= 0.55  →  High
  PCIS <  0.55  →  Moderate

9.6 FEATURE IMPORTANCE
  - Stored in pcis.joblib after training
  - Exposed via API: GET /api/v1/analytics/feature-importance
  - Shows which factors drive congestion impact most

ACTUAL RESULTS (after rescore):
  - Severe hotspots:  13
  - Critical hotspots: 23
  - Top PCIS: 1.000 (Elite Junction)
  - Average PCIS: 0.603
  - PCIS R²: 0.9524
  - PCIS MAE: 0.0133

10. TEMPORAL FORECASTING MODEL
================================================================================

FILE: backend/app/services/forecast.py

PURPOSE:
  Predict violation counts by hour to enable proactive patrol scheduling.

APPROACH: Ensemble of LightGBM + Hour Lookup Table

10.1 TRAINING DATA
  - Aggregates city-wide violations by (hour_ist, day_of_week)
  - ~168 unique hour-day combinations

10.2 LIGHTGBM MODEL
  - Features: hour_ist, day_of_week, month, is_weekend
  - n_estimators=200, learning_rate=0.06, max_depth=4

10.3 HOUR LOOKUP TABLE
  - Historical mean violations per (hour, day_of_week) combination
  - Provides stable baseline predictions

10.4 ENSEMBLE PREDICTION
  - Final = 25% LightGBM + 75% Lookup Table
  - Evaluated on stable hours (above median violation count)

10.5 PER-HOTSPOT FORECAST
  - Scales city prediction by hotspot's share of total violations
  - Returns 24-hour forecast with confidence intervals (±1.96σ)
  - Exposed via: GET /api/v1/hotspots/{id}/forecast

ACTUAL RESULTS:
  - Forecast MAPE: 24.83%
  - Forecast RMSE: 32.469
  - Peak violation hour (IST): ~10:00 AM (morning commercial rush)


================================================================================
11. ENFORCEMENT RECOMMENDATION ENGINE
================================================================================

FILE: backend/app/services/recommender.py

PURPOSE:
  Generate ranked patrol deployment plans for traffic police.

11.1 PRIORITY RANKING
  - Sorts all hotspots by PCIS descending
  - Selects top N zones based on available officers parameter

11.2 TIME WINDOW RECOMMENDATION
  - temporal_congestion >= 0.65  →  08:30–11:00 IST (morning peak)
  - temporal_congestion >= 0.45  →  17:00–20:00 IST (evening peak)
  - Otherwise                    →  10:00–13:00 IST (midday)

11.3 TEAM ALLOCATION
  - PCIS >= 0.80 OR violations >= 500  →  3 officers
  - PCIS >= 0.65 OR violations >= 250  →  2 officers
  - Otherwise                          →  1 officer

11.4 TOW TRUCK ASSIGNMENT
  - Assigned if spillover_risk >= 0.55 and tow trucks available
  - Limited by tow_trucks parameter (default 2–3)

11.5 DELAY SAVINGS ESTIMATE
  - delay_saved = estimated_delay_min_per_hour × (0.55 + PCIS × 0.35)

11.6 RATIONALE GENERATION
  - Auto-generated text explaining why zone is prioritized
  - Includes dominant violation type, junction impact %, spillover risk %

OUTPUT:
  - enforcement_plan.json — Saved patrol plan
  - API: GET /api/v1/enforcement/plan?officers=6


================================================================================
12. WHAT-IF SIMULATION MODULE
================================================================================

FILE: backend/app/services/recommender.py (simulate_deployment function)

PURPOSE:
  Allow command center to simulate impact of deploying extra enforcement teams.

INPUTS:
  - extra_teams: Number of additional patrol teams (1–10)
  - duration_hours: Deployment duration (1–8 hours)

LOGIC:
  - Takes top 10 PCIS hotspots as baseline
  - efficiency = min(0.25 + extra_teams × 0.12, 0.72)
  - projected_reduction = baseline_delay × efficiency × duration_hours
  - projected_avg_pcis = avg_pcis × (1 - efficiency × 0.45)

OUTPUTS:
  - baseline_delay_min_per_hour
  - projected_delay_reduction_min
  - projected_avg_pcis_after
  - top_zones_covered (list of zone names)

API: GET /api/v1/simulate?extra_teams=2&duration_hours=2

  - forecast_rmse          — Root Mean Square Error

    MAPE <= 18%         → +2 pts  |  <= 28%   → +1 pt

  Grade assignment:
    5+ points  →  A+ Production Ready
    3–4 points →  A Pilot Ready
    < 3 points →  B Prototype

22. INNOVATION HIGHLIGHTS (FOR JUDGES)
================================================================================

1. PCIS (Parking Congestion Impact Score)
   — Novel metric linking parking violations to estimated traffic delay,
     not just challan counts. Goes beyond simple violation heatmaps.

2. Hybrid Hotspot Detection
   — Combines named junction intelligence (40 BTP junctions) with
     data-driven corridor detection (20 H3 zones). Best of both worlds.

3. Real Data at Scale
   — 298,450 actual Bengaluru violations. Not synthetic or demo data.
     Top hotspots match known city congestion pain points.

4. ML Stack with Published Metrics
   — LightGBM + Isotonic Calibration for PCIS (R² 0.95)
   — Ensemble forecast (MAPE 24.8%)
   — Silhouette-validated clustering
   — Transparent model grading system

5. Decision Support, Not Just Analytics
   — Enforcement plan generator with time windows and team sizes
   — What-If simulator quantifying delay reduction per deployment
   — Actionable output for traffic police command center

6. End-to-End Deployable System
   — Single command to train, single command to serve
   — Working dashboard at localhost:8000
   — 12 REST API endpoints for integration with existing systems

7. Zero New Hardware Required
   — Works entirely with existing police violation data
